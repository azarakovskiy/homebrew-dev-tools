---
description: Use when a task or spec is incomplete, ambiguous, contradictory, not testable, or missing acceptance criteria. Do not use once the spec is already implementation-ready.
---

Refine the spec into something implementation-ready.

Steps:
1. Extract goal, scope, constraints, and non-goals.
2. Identify ambiguity, contradictions, and unstated assumptions.
3. Convert vague requirements into observable behavior.
4. Add missing edge cases and failure cases.
5. Write explicit acceptance criteria.
6. Define "done when" in a testable way.
7. Keep scope minimal.

Output:
- gaps
- questions / proposed resolutions
- acceptance criteria
- done when
